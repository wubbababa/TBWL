【严重 / 生产阻断 — 最高优先级】
P0 安全问题：登录守卫看似完成，实际未真正闭环
------------------------------------------------------------
S1 — orders 表 RLS 被直接关闭，数据对公网裸奔  ⚠️ 必须修复
  现状：fix_orders_rls.sql 采用「方案 A」ALTER TABLE orders DISABLE ROW
        LEVEL SECURITY。前端用公开的 anon key 浏览器端直连 Supabase，
        关闭 RLS 后任何人拿 anon key 调 Supabase REST API 即可读写/删除
        全部订单，完全绕过 proxy.ts 登录守卫。
  修复：改用文件中注释掉的「方案 B」——保留 RLS，新增 authenticated 全量
        策略（FOR ALL TO authenticated USING(true) WITH CHECK(true)）。

S2 — proxy.ts 守卫可被伪造 cookie 绕过  ⚠️ 必须修复
  现状：仅检查 cookie 名以 sb- 开头、以 -auth-token 结尾且非空，不校验
        JWT 真伪。手动塞一个 sb-x-auth-token=abc 即可进入页面框架。
  修复：升级为真实校验 supabase.auth.getUser(token)；真正防线仍是 RLS
        （S1 修好后伪造者也只能看到空表）。

S3 — /api/account-stats 无任何认证  ⚠️ 必须修复
  现状：GET 完全裸露，任何人可拉取账户余额/累计消费/累计充值汇总。
  修复：加上与 create-user 一致的 Bearer 会话校验。

S4 — /api/admin/create-user 缺角色校验  ⚠️ 必须修复
  现状：只校验「调用者是已登录用户」，不校验是否管理员，导致任何普通登录
        用户都能创建新账号。
  修复：引入角色概念（如 user_metadata.role === 'admin'）后再放行。
